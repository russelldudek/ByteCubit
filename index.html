Four obligations before cutover.

A modernization engineer should make each obligation explicit. That is how the team avoids replacing one opaque system with another.

01

Read the live function

Inventory tags, states, SQL logic, historian dependencies, user decisions, failure behavior, and support ownership before designing the replacement.

02

Build the target twin

Develop the Ignition function as a reusable application component with explicit state, named interfaces, testable logic, and visible operational diagnostics.

03

Reconcile context

Normalize naming, timestamps, units, asset hierarchy, metadata, and SQL/PI relationships so the data means the same thing to production, quality, R&D, and analytics.

04

Earn the release

Prove parity, performance, exception handling, operator usability, ownership, rollback, and post-cutover observation before retiring the legacy path.

Measure modernization as an operating outcome.

No invented maturity score. The useful comparison is current behavior versus verified target behavior, using measures the team can actually observe.

First 90 days: stabilize, twin, release.

The contract needs useful production movement quickly, but the first visible deliverable should also install a repeatable modernization method.

Days 1-30

Read and stabilize

  • Map applications, tags, SQL, PI assets, users, dependencies, and support paths.
  • Select one bounded high-value function.
  • Establish naming, versioning, test, and release conventions.
  • Resolve urgent data-quality and supportability risks.
Days 31-60

Build the twin

  • Develop the Ignition target component.
  • Run legacy and target behavior in parallel where feasible.
  • Reconcile SQL and historian context.
  • Test operator states, exceptions, performance, and rollback.
Days 61-90

Release and compound

  • Cut over the proven function with a defined observation window.
  • Publish reusable templates, runbooks, and acceptance criteria.
  • Prioritize the next modernization tranche.
  • Define the cloud data contract and ownership model.

Questions I would bring into the room.

The answers determine whether this is mainly an Ignition build, a data-estate repair, or a broader manufacturing application operating-model reset.

Which legacy MES functions are business-critical, and which are simply familiar?
Where do current users compensate for system gaps with spreadsheets, manual entries, or local knowledge?
What is the current Ignition architecture, module mix, deployment pattern, and version?
How are tag standards, UDTs, projects, environments, source control, testing, and release authority managed?
Which SQL and PI consumers would be harmed by a naming, timestamp, or asset-model change?
What does “clean up PI” mean here: duplicates, stale points, naming, AF structure, interfaces, ownership, or all of the above?
Which cloud data-lake consumers need raw events, contextualized assets, or curated operational products?
What must be demonstrably true by year-end for the extension decision to be obvious?

Modernization should leave the plant easier to understand.

That means better applications, cleaner data context, clearer ownership, safer cutovers, and a method the next engineer can reuse.

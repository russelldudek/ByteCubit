Interview Thesis Brief

Industrial Automation Engineer (Ignition Development) · Bytecubit Technologies

1 / 4

Executive thesis

Don't migrate screens. Prove the signal path.
Run legacy and target behavior side by side, reconcile data context, and cut over only when function, data, ownership, and rollback evidence are explicit.

Actual mandate

Modernize legacy MES functionality with Ignition while protecting production continuity; improve SQL-backed manufacturing applications; standardize the AVEVA PI historian environment; and create dependable manufacturing data integration into a cloud data lake.

Why this matters now

Legacy functionality

Business-critical behavior may be encoded across screens, queries, scripts, workarounds, and local knowledge.

Historian entropy

Duplicate, stale, poorly named, or weakly contextualized signals increase support burden and reduce analytic trust.

Cloud demand

Raw plant signals are not automatically usable data products; context, ownership, quality, and semantics must travel with them.

Contract clock

The team needs visible progress through year-end without sacrificing maintainability for speed.

Central contradictions

  • Fast delivery vs. production continuity.
  • Reusable Ignition standards vs. legitimate local process differences.
  • Historian cleanup vs. unknown downstream dependencies.
  • Cloud accessibility vs. OT reliability, ownership, and security.

Illustrative scenario states

Legacy function

Focus: hidden behavior and consumers. Minimum evidence: state/event/context/rollback parity.

Historian cleanup

Focus: point usage, stewardship, AF context, retention, and consumer impact before deletion or renaming.

New line onboarding

Focus: reusable tag/UDT/application templates without scaling an unproven local pattern.

Cloud analytics feed

Focus: freshness, semantics, quality, access, and named data-product ownership.

Release criteria

Function parity · Data integrity · Operator usability · Performance · Security/access · Support ownership · Rollback readiness · Post-release observation.

Interview Plan

Use the interview to test the work, the architecture, and the operating fit.

4 / 4

Working-session proposal

  1. Select a representative legacy function or anonymized SQL/tag path.
  2. Map the live states, dependencies, consumers, and failure modes.
  3. Sketch the target Ignition component, tag/UDT structure, database interaction, and diagnostics.
  4. Define parity, performance, exception, usability, rollback, and observation tests.
  5. Identify what should be standardized and what must remain plant-specific.

Executive questions

  • Which functions must be modernized by year-end, and why those first?
  • What is the current Ignition architecture and deployment discipline?
  • Where is ownership ambiguous across manufacturing, controls, IT, data, R&D, and vendors?
  • Which PI cleanup decisions are reversible, and which require a consumer-impact map?
  • What does the cloud data lake need to enable that current plant systems cannot?
  • What would make a contract extension an obvious business decision?

Success measures

Validated function releases; production disruption avoided; recurring support issues reduced; cleaner data context; faster diagnosis; reusable templates; accountable owners; accepted cloud data products; learning captured for the next plant or function.

Closing position

The role should leave more than working code. It should leave a more legible manufacturing system: explicit behavior, trustworthy data, known ownership, safer releases, and a repeatable modernization method.
